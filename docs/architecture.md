# 시스템 아키텍처

## 1. 선택 방향

초기 시스템은 하나의 저장소와 하나의 배포 단위 안에서 도메인 경계를 분명히 하는 **모듈형 모놀리스**로 구축한다. 답안 저장과 제출에 필요한 트랜잭션 일관성을 단순하게 유지하고, 운영 복잡도는 실제 규모가 요구할 때만 증가시킨다.

기술 스택은 팀 역량과 운영 플랫폼을 확인한 뒤 ADR로 확정한다. 특정 프레임워크보다 아래 경계와 계약을 우선한다.

## 2. 논리 구성 요소

```text
[Candidate Web] ─┐
                 ├── HTTPS ── [API Application] ── [Relational Database]
[Admin Web] ─────┘                  │
                                    └── [Background Worker]
```

| 구성 요소 | 책임 |
| --- | --- |
| Candidate Web | 문제 표시, 로컬 상태, 자동 저장, 타이머, 접근성 |
| Admin Web | 시험·문제·배정 관리, 진행 현황, 운영 조치 |
| API Application | 인증·인가, 도메인 규칙, 트랜잭션, 응답 모델 |
| Background Worker | 만료 응시 복구, 알림, 내보내기 등 비동기 작업 |
| Relational Database | 시험, 응시, 답안, 제출, 감사 이벤트의 원장 |

Worker는 초기에도 동일 코드베이스의 별도 실행 모드로 둘 수 있다. 메시지 큐는 작업량과 신뢰성 요구를 측정한 뒤 도입한다.

## 3. API 내부 모듈

- `identity`: 인증 주체, 역할, 권한 검사
- `exam`: 시험 일정, 정책, 문제와 선택지
- `assignment`: 응시자 배정과 입장 자격
- `attempt`: 응시 세션 생성, 상태 전이, 시간 판정
- `answer`: 답안 검증, 리비전, 멱등 저장
- `submission`: 최종·자동 제출과 제출 결과
- `audit`: 보안·운영 이벤트 기록 및 제한된 조회

모듈은 다른 모듈의 테이블을 임의로 수정하지 않고 공개된 애플리케이션 서비스 또는 명시적인 도메인 이벤트를 통해 협력한다.

## 4. 초기 API 계약

| 메서드 및 경로 | 목적 | 핵심 규칙 |
| --- | --- | --- |
| `GET /api/v1/exams/assigned` | 배정 시험 조회 | 로그인 사용자 범위만 반환 |
| `POST /api/v1/exams/{examId}/attempts` | 응시 시작 | 활성 응시 중복 방지, 마감 계산 |
| `GET /api/v1/attempts/{attemptId}` | 응시 상태 복원 | 소유권 검사, 서버 시각 반환 |
| `GET /api/v1/attempts/{attemptId}/questions` | 응시 문제 조회 | 정답·채점 규칙 제외 |
| `GET /api/v1/attempts/{attemptId}/answers` | 저장 답안 조회 | 최신 서버 리비전 반환 |
| `PUT /api/v1/attempts/{attemptId}/answers/{questionId}` | 답안 저장 | 멱등성, 리비전, 마감 검사 |
| `POST /api/v1/attempts/{attemptId}/submission` | 최종 제출 | 정확히 하나의 제출 결과 보장 |
| `GET /api/v1/attempts/{attemptId}/submission` | 제출 상태 조회 | 제출 식별자와 서버 시각 반환 |

모든 변경 API는 인증, CSRF 또는 토큰 보호, 자원 소유권, 역할, 응시 상태, 감사 정책을 공통 적용한다.

## 5. 데이터 경계

- 응시자 조회 모델에는 문제 본문, 선택지, 배점 등 응시에 필요한 정보만 포함한다.
- 정답, 해설, 채점 규칙은 별도 권한이 필요한 관리·채점 경계에 둔다.
- 답안과 제출은 관계형 트랜잭션으로 처리한다.
- 감사 이벤트는 일반 애플리케이션 수정 흐름에서 갱신·삭제할 수 없도록 제한한다.
- 정책은 응시 시작 시 스냅샷으로 고정해 진행 중 설정 변경의 영향을 차단한다.

## 6. 배포와 확장

### 초기

- 단일 리전
- API 다중 인스턴스 가능, 세션 상태는 서버 메모리에 의존하지 않음
- 관리형 관계형 데이터베이스와 자동 백업
- 정적 웹 자산은 CDN 배포 가능
- 구조화 로그, 메트릭, 분산 추적 식별자 적용

### 확장 판단 기준

다음 조건을 측정한 후에만 구성 요소를 추가한다.

- 읽기 부하가 데이터베이스 병목으로 확인되면 제한적 캐시 검토
- 만료·내보내기 작업이 API 지연에 영향을 주면 큐와 Worker 확장
- 특정 모듈의 독립 배포 필요성과 운영 이득이 분리 비용보다 클 때 서비스 분리
- 응답 또는 파일이 시스템 메모리의 10% 이상이면 스트리밍 처리 검토

## 7. 주요 기술 결정 기록 대상

1. 프론트엔드·백엔드 프레임워크와 지원 버전
2. 인증 방식과 세션 수명
3. 브라우저 임시 저장소 및 암호화 정책
4. 데이터베이스와 마이그레이션 도구
5. 멱등성 키 저장 기간
6. 자동 제출 실행 방식
7. RPO, RTO, 백업 복구 절차
8. 예상 동시 응시자 수와 용량 산정
