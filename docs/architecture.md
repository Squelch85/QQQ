# 시스템 아키텍처

## 1. 선택 방향

초기 시스템은 온라인 서버 없이 한 실행 단위 안에서 시험지 불러오기, 응시, 제출, 평가, 채점을 처리하는 **로컬 단일 애플리케이션**으로 구축한다. 데이터베이스, 인증 서버, 백그라운드 작업, 네트워크 동기화는 도입하지 않는다.

기술 스택은 지원 운영체제와 배포 방식을 확인한 뒤 ADR로 확정한다. 특정 프레임워크보다 단순한 데이터 흐름과 결정적인 채점 결과를 우선한다.

## 2. 논리 구성 요소

```text
[시험지 파일] ──> [로컬 응시 애플리케이션] ──> [화면 결과]
                         │
                         └──────────────> [로컬 누적 저장/CSV/인쇄]
```

| 구성 요소 | 책임 |
| --- | --- |
| 시험지 로더 | 파일 읽기, 스키마 검증, 도메인 객체 변환 |
| 응시 화면 | 문제 표시, 답안 입력, 문제 이동, 접근성 |
| 로컬 상태 저장소 | 제출 전 답안은 메모리에 유지하고 제출 결과는 시험 ID별로 브라우저 저장소에 누적 |
| 타이머 | 경과 시간 계산, 시간 경고, 만료 동작 실행 |
| 채점 엔진 | 문제 유형별 평가, 점수 합산, 검토 필요 분류 |
| 결과 화면 | 총점, 문항별 결과, 정답·해설 공개 정책 적용 |
| 결과 내보내기 | 제출 직후 최신 누적 CSV를 생성하고 필요 시 다시 저장하거나 인쇄 |

## 3. 내부 모듈

- `exam-loader`: 시험지 파일 파싱과 스키마 검증
- `exam`: 시험, 문제, 선택지, 채점 규칙
- `attempt`: 시작, 진행, 제출, 종료 상태 전이
- `answer`: 문제 유형별 답안 검증과 메모리 상태 갱신
- `grading`: 문항 평가, 점수 계산, 결과 생성
- `report-storage`: 시험 ID별 응시 결과 누적 저장
- `result-export`: 누적 CSV 생성과 인쇄 모델

모듈은 명시적인 입력과 반환값으로 협력한다. 채점 모듈은 UI 상태나 파일 시스템에 직접 의존하지 않는 순수 로직으로 유지한다.

## 4. 애플리케이션 내부 계약

| 작업 | 입력 | 출력 | 핵심 규칙 |
| --- | --- | --- | --- |
| `loadExam` | 시험지 파일 | 검증된 `Exam` 또는 오류 목록 | 필수 필드와 정답·배점 일관성 검사 |
| `startAttempt` | 검증된 `Exam` | `Attempt` | 상태를 `READY`에서 `IN_PROGRESS`로 전환 |
| `setAnswer` | 응시, 문제 ID, 답안 | 갱신된 답안 상태 | 진행 중이고 문제 유형에 맞을 때만 변경 |
| `submitAttempt` | 응시와 현재 답안 | `Submission` | 답안 스냅샷 생성 후 추가 변경 금지 |
| `gradeSubmission` | 시험, 제출 | `GradeResult` | 동일 입력에 대해 결정적인 결과 반환 |
| `exportResult` | 채점 결과, 선택 메타데이터 | 파일 또는 인쇄 모델 | 명시적으로 요청한 정보만 포함 |

## 5. 데이터 경계

- 시험지 파일에는 문제, 선택지, 정답, 배점, 공개 가능한 해설이 포함될 수 있다.
- 응시 화면 모델은 제출 전 정답과 채점 규칙 필드를 참조하거나 렌더링하지 않는다.
- 현재 답안은 기본적으로 메모리에만 유지한다.
- 제출 스냅샷과 채점 결과는 시험 ID별로 브라우저 저장소에 누적하고 제출 직후 최신 누적 CSV로 기록한다.
- 결과 파일에는 기본적으로 최소 정보만 포함하고 응시자 이름이나 식별자는 선택적으로 추가한다.

정답이 같은 파일에 포함되므로 이 도구는 강한 비밀 보호나 부정행위 통제가 필요한 고위험 시험을 대상으로 하지 않는다. 필요한 경우 향후 시험지 패키지 암호화 또는 별도 작성 도구를 검토한다.

## 6. 배포와 확장

### 초기

- 단일 기기에서 실행
- 네트워크 연결 불필요
- 서버, 데이터베이스, 계정, 운영 대시보드 없음
- 시험지 파일과 애플리케이션만 배포
- 결과 보관은 브라우저 자동 누적과 제출 시 CSV 자동 저장을 기본으로 하며 다시 저장·인쇄를 지원

### 확장 판단 기준

다음 요구가 실제로 확인된 경우에만 기능을 추가한다.

- 브라우저 새로고침이나 종료 복구 요구가 있으면 로컬 임시 저장 검토
- 시험지 배포 보호 요구가 있으면 서명 또는 암호화 패키지 검토
- 수동 검토 문항이 많아지면 별도 채점 화면 검토
- 다중 기기 집계가 필요해지면 결과 파일 가져오기 기능을 우선 검토
- 온라인 중앙 관리가 필수가 되면 현재 오프라인 제품과 분리된 확장 단계로 설계
- 대표 시험지에서 측정된 병목이 있을 때만 자료구조나 처리 방식을 최적화

## 7. 주요 기술 결정 기록 대상

1. 데스크톱, 로컬 웹 또는 단일 실행 파일 중 배포 방식
2. 지원 운영체제와 브라우저
3. 시험지 파일 형식과 스키마 버전
4. 정답·해설의 제출 후 공개 정책
5. 단답형 정규화와 부분 점수 규칙
6. 로컬 타이머와 시간 만료 동작
7. 결과 파일 형식과 개인정보 포함 정책
8. 선택적 임시 저장과 파일 보호 방식
