# 품질 및 성능 전략

## 1. 품질 목표

우선순위는 **자격 판정 정확성 > 원본 결과 불변성 > 인증서 발행 통제 > 개인정보 최소화 > 사용 단순성 > 성능** 순서로 둔다. 성능 향상을 위해 평가 결과의 정확성, 결과 불변성, 발행 차단 조건을 희생하지 않는다.

## 2. 현재 자동화된 검증

현재 구현은 필기시험 흐름, 기존 결과·인증서 흐름, SQLite 기반 R&R·교육·자격 판정 API를 자동 검증한다.

```bash
npm test
npm run check
```

현재 자동화 범위는 다음을 포함한다.

- 필기시험 스키마와 필수 필드 검증
- 필기시험 응시 상태와 제출 스냅샷
- 단일 선택, 복수 선택, 단답형 채점
- 리포트 CSV 생성과 파싱
- 브라우저 저장소와 디렉터리 기반 리포트 저장 보조 로직
- 인증서 UI와 결과 API의 기존 동작
- `server.py` 단위 테스트와 Python 문법 검사
- `src/standalone.js` 빌드와 주요 JavaScript 파일 문법 검사

`npm test`와 `npm run check`는 필기시험 흐름, SQLite 스키마, R&R 계산, 교육 검증, 통합 자격 판정, 인증서 이력 경로가 깨지지 않았다는 회귀 확인이다.

## 3. 도메인 단위 테스트

- 필기시험 점수와 항목별 과락 판정의 SQLite 저장
- 제출 후 답안 원본 변경 차단
- 계수형 R&R 전체 일치율, OK 일치율, NG 검출률 계산
- 계수형 R&R 1종 오류율, 2종 오류율, 반복 일치율, 불량 유형 일치율 계산
- 계량형 R&R 측정값 누락과 숫자 형식 오류 검증
- 계량형 R&R EV, AV, GRR, %GRR, ndc 계산
- 교육 이수 시간과 증빙 충족 여부 판정
- `validateCertificationReadiness()`의 발행 가능 여부 판정
- 감사 로그의 민감 필드 제거와 참조값 저장

## 4. 통합 테스트

- 필기시험 제출 -> SQLite 저장 -> 채점 결과 저장
- 계수형 R&R 제출 -> 결과 계산 -> SQLite 저장
- 계량형 R&R CSV import -> 검증 -> 계산 -> SQLite 저장
- 교육 기록 등록 -> 자격 판정 반영
- 필수 평가 누락 -> 정식 인증서 발행 차단
- 승인 완료 -> 정식 인증서 발행과 해시 저장
- DB 장애 -> `LOCAL_ONLY` 출력물 생성 시 정식 인증서와 명확히 구분
- 인증서 재발행 -> 기존 이력 보존
- CSV export가 SQLite 원본에서 파생되는지 확인

## 5. E2E 테스트 범위

- 외관검사자 자격 인증 전체 흐름: 필기시험, 사진 기반 계수형 R&R, 교육 이수, 자격 판정, 정식 인증서 발행
- 수입검사자 자격 인증 전체 흐름: 필기시험, 계량형 R&R, 교육 이수, 자격 판정, 정식 인증서 발행
- 필수 R&R 누락 상태에서 정식 인증서 발행 차단
- 교육 증빙 누락 상태에서 정식 인증서 발행 차단
- 인증서 발행 후 응시자별, 자격별, 인증서별 조회

## 7. 성능 지침 적용 범위와 우선순위

모든 신규 코드와 리팩토링은 이 성능 지침을 적용한다. 다만 우선순위는 항상 **자격 판정 정확성 > 원본 결과 불변성 > 인증서 발행 통제 > 개인정보 최소화 > 사용 단순성 > 성능**이다. 최적화는 측정으로 증명된 경우에만 수행하며, 불필요한 미세 최적화로 기능 정확성이나 결과 일관성을 훼손하지 않는다.

## 8. 초안 단계 성능 기준

- 동일 연산이 반복되면 변수로 캐싱해 중복 계산을 줄인다.
- 파일 재열기와 중복 접근을 피한다. 소규모 입력은 가독성을 위해 단순 로드를 허용하되, 반복 I/O는 금지한다.
- 단회 순회와 간단 변환은 제너레이터를 우선 검토한다. 디버깅이나 재사용 때문에 리스트를 유지하면 PR 본문에 사유를 1줄 기록한다.
- 변경이 성능에 영향을 줄 가능성이 있으면 5회 이상 측정하고 중앙값과 IQR을 기록한다.

## 9. 리뷰 전 안정화 기준

| 사용 패턴 | 권장 구조 | 적용 기준 |
| --- | --- | --- |
| 멤버십 탐색 빈번 | `set` 또는 `dict` | `N≥1e4` |
| 양끝 삽입·팝 중심 | `deque` | 양끝 삽입·팝 비율 `≥70%` |
| Top-K 추출 | `heapq` | `K≤N/20` |

- 단순 매핑은 `dict`를 검토하고, 상태 변경이나 부작용이 있는 로직은 명시적 `if`를 유지한다.
- 반복 정규식은 `re.compile`을 검토하고, 루프에서 반복 참조되는 상수와 함수는 루프 밖 바인딩을 검토한다. 1만 회 미만 호출 경로는 가독성을 우선해 생략할 수 있다.
- 주요 성능 PR은 `npm run benchmark:report` 같은 단순 스크립트 로그나 별도 벤치마크 결과를 제출한다.

## 10. 병목 대응 및 대용량 처리 기준

- 파일 또는 응답 크기가 시스템 메모리의 10% 이상이면 스트리밍 처리를 검토한다. Python JSON 대용량 처리는 필요 시 `orjson` 또는 `ijson` 도입을 검토한다.
- 병렬화는 프로파일로 병목 원인을 확인한 뒤 적용한다. CPU-bound는 `multiprocessing` 또는 `joblib`, I/O-bound는 `asyncio` 또는 `ThreadPoolExecutor`를 검토하며, 순이득 20% 이상이 예상될 때만 도입한다.
- 객체가 10만 개 이상이고 동적 속성이 필요 없으면 Python 모델에 `@dataclass(slots=True)` 적용을 검토한다.
- 바이트 핫패스에 한해 `memoryview` 또는 `bytearray`를 검토한다.
- 반복 수치 연산은 `numpy`, `numexpr`, `numba` 같은 벡터화 도구를 검토하되, 외부 의존성 추가 시 별도 승인 사유를 기록한다.

## 11. 성능 검증 및 품질 게이트

- 변경 전후 동일 환경과 동일 입력으로 비교한다.
- 5~10회 반복 후 중앙값과 IQR을 기록한다. 필요 시 Mann-Whitney U 같은 비모수 통계를 사용한다.
- 중앙값 기준 10% 이상 악화되면 수정 요청으로 판단한다. 단, 기능 정확성·보안·불변성 확보를 위한 의도된 비용이면 승인 사유를 기록한다.
- 단순 p95 비교만으로 반려하지 않는다.
- 입력 크기는 기본적으로 `N`, `10N` 2단계를 사용하고, 대용량 문제가 확인되면 `100N`까지 확장한다.
- 시간과 메모리 중 2항목 이상을 첨부한다. 메모리 측정이 환경상 어렵다면 측정 불가 사유를 적고 시간·처리량·파일 크기 같은 대체 지표를 사용한다.
- 프로파일링이 필요한 경우 상위 병목 함수 3~5개 요약 또는 Flamegraph 1장을 첨부한다.

## 12. 벤치마크 실행

현재 리포트 CSV 생성·파싱 경로는 다음 명령으로 측정한다.

```bash
npm run benchmark:report
```

출력은 입력 건수, CSV byte 크기, 생성 중앙값/IQR, 파싱 중앙값/IQR을 포함한다. 성능 PR은 이 로그 또는 동일한 형식의 재현 가능한 로그를 PR 본문에 첨부한다.

## 13. PR 및 리뷰 작성 지침

- 모든 PR 제목, 리뷰, 사고기록은 한국어로 작성한다.
- 성능 PR 제목은 `[perf] 변경 요약` 형식을 사용한다. 예: `[perf] 파일 스트리밍 도입`, `[perf] dict 기반 분기 단순화`
- PR 본문은 다음 항목을 포함한다.
  - **배경/문제:** 병목 원인과 영향 범위
  - **변경 내용:** 적용 기법과 코드 구조 변화
  - **결과:** 전/후 중앙값, IQR, 필요 시 메모리·스루풋
  - **호환성:** API 변경 여부, 기본값 안정성, 롤백 용이성
- 리뷰 판단은 수치와 복잡도 영향에 근거한다. 예: “중앙값 기준 성능 +18% 향상, 복잡도 영향 없음 → 승인”
