# 품질 및 성능 전략

## 1. 품질 목표

우선순위는 **채점 정확성 > 답안 보존 > 사용 단순성 > 접근성 > 개인정보 최소화 > 성능 > 부가 기능** 순서로 둔다. 성능 향상을 위해 평가 결과의 정확성이나 일관성을 희생하지 않는다.

## 2. 테스트 피라미드

### 단위 테스트

- 시험지 스키마와 필수 필드 검증
- 응시 상태 전이와 금지 전이
- 문제 유형별 답안 값 검증
- 단일·복수 선택 및 단답형 채점
- 미응답, 부분 점수, 검토 필요 처리
- 문항 점수 합산과 만점 상한
- 타이머 경과와 시간 만료 정책

### 통합 테스트

- 파일 로더에서 도메인 모델 변환
- 답안 입력에서 제출 스냅샷 생성
- 제출 이후 답안 변경 차단
- 제출 스냅샷에서 채점 결과 생성
- 정답 공개 정책에 따른 결과 모델 생성
- 결과 파일 저장과 개인정보 필드 선택
- 잘못된 파일 입력 시 안전한 오류 처리

### E2E 테스트

- 시험지 열기, 시작, 응답, 제출, 결과 확인
- 객관식 단일·복수 선택 혼합 시험 채점
- 단답형 정규화와 허용 답안 채점
- 미응답 문항 경고 후 제출
- 시간 만료 후 제출 또는 자동 제출
- 검토 필요 문항이 포함된 결과 표시
- 결과 파일 저장과 인쇄
- 네트워크가 없는 환경에서 전체 흐름 완료

## 3. 접근성 검증

- 키보드만으로 시험 시작, 문제 이동, 답안 선택, 제출 가능
- 포커스 순서와 포커스 표시가 명확함
- 응답 여부, 오류, 시간 경고, 채점 완료가 스크린 리더에 전달됨
- 색상만으로 정오나 미응답 상태를 구분하지 않음
- 확대 및 작은 화면에서도 문제와 조작 영역이 손실되지 않음
- 제한 시간 비활성화 또는 연장 설정을 시험지 정책으로 제공 가능

## 4. 품질 지표

| 지표 | 목표 방향 |
| --- | --- |
| 고정 테스트 벡터 채점 불일치 | 0건 |
| 제출 후 답안 변경 가능 건수 | 0건 |
| 유효 시험지 불러오기 성공률 | 100% |
| 잘못된 시험지의 비정상 종료 | 0건 |
| 오프라인 전체 흐름 성공률 | 100% |
| 접근성 핵심 흐름 실패 | 0건 |
| 결과 파일의 비선택 개인정보 포함 | 0건 |

온라인 서비스 성공률, API 지연시간, 동시 접속자 수, 재접속 복구율은 측정 대상이 아니다.

## 5. 대표 성능 시나리오

1. **시험지 불러오기:** 문제와 선택지가 포함된 로컬 파일을 파싱하고 검증한다.
2. **문제 이동:** 답안 상태를 유지하며 이전·다음 문제를 반복 표시한다.
3. **일괄 채점:** 제출 후 모든 문항을 평가하고 점수를 합산한다.
4. **결과 생성:** 문항별 결과와 선택적 결과 파일을 생성한다.

입력 규모는 대표 시험지 `N`과 대형 시험지 `10N`을 기본으로 한다. 일반 사용 규모가 작으므로 스트리밍, 병렬화, 별도 인덱스는 실측 병목이 확인되기 전에는 도입하지 않는다.

## 6. 성능 측정 원칙

- 변경 전후 동일 환경과 동일 시험지로 비교한다.
- 최소 5회, 권장 10회 반복한다.
- 중앙값과 IQR을 기록하고 필요하면 최대 메모리와 처리 문항 수를 함께 측정한다.
- 중앙값 기준 10% 이상 악화되면 원인을 분석하고 수정 또는 승인 사유를 기록한다.
- 단순 단일 실행시간만으로 변경을 반려하지 않는다.
- 프로파일링 시 상위 병목 함수 3~5개 또는 Flamegraph를 첨부한다.
- 실제 병목이 확인되기 전에는 병렬화, 캐시, 특수 자료구조를 도입하지 않는다.

현재 단계는 실행 코드와 대표 시험지 데이터가 없으므로 성능 측정을 수행하지 않는다. 이번 문서 변경은 런타임 성능에 영향을 주지 않으며 온라인 복잡성을 제거하는 가독성과 범위 명확화를 우선한다.

## 7. 진단 가능성

- 사용자에게 시험지 검증 오류의 위치와 수정 방법을 표시한다.
- 내부 오류는 앱 버전, 오류 코드, 동작 단계만 기록한다.
- 답안 원문, 정답, 해설, 응시자 식별 정보는 로그에 포함하지 않는다.
- 채점 오류 재현을 위해 개인정보가 제거된 최소 시험지와 답안 테스트 벡터를 사용할 수 있게 한다.
- 온라인 대시보드, 추적 ID, 서버 경보는 두지 않는다.

## 8. 출시 게이트

- 핵심 인수 조건 자동화 완료
- 고정 채점 테스트 벡터 전부 통과
- 제출 이후 답안 불변성 검증
- 네트워크 차단 환경의 E2E 흐름 통과
- 잘못된 시험지와 파일 경계값 테스트 통과
- 키보드 응시와 주요 접근성 흐름 검증
- 결과 파일 개인정보 기본값 검증
- 대표 시험지 규모에서 합의된 반응성 충족


## 누적 리포트 성능 게이트

`node tools/benchmark-report.mjs`로 1,000건과 10,000건 입력을 각각 7회 측정하고 CSV 생성·파싱 중앙값과 IQR, 직렬화 바이트를 기록한다. 중앙값이 기준 대비 10% 이상 악화하면 원인을 검토한다. 현재는 브라우저 로컬 저장소 구조를 유지하며, 대표 운영 규모에서 측정된 병목이나 용량 문제가 확인될 때만 IndexedDB 전환을 검토한다.

2026-06-08 기준 측정 결과는 다음과 같다.

| 입력 | CSV 크기 | 생성 중앙값 (IQR) | 파싱 중앙값 (IQR) |
| --- | ---: | ---: | ---: |
| 1,000건 | 156,794 bytes | 10.44ms (7.93ms) | 26.15ms (23.70ms) |
| 10,000건 | 1,595,324 bytes | 75.59ms (28.34ms) | 199.28ms (54.30ms) |

10,000건에서도 직렬화 크기가 2MB 미만이고 파싱 중앙값이 200ms 수준이므로, 복잡도를 높이는 IndexedDB 전환은 보류한다. 대신 1,000건 또는 2MB부터 사용자에게 CSV 백업과 기록 정리를 안내한다.

## 인원별 CSV I/O 최소화

OPFS 인원별 디렉터리는 `candidates/index.json`에 파일별 마지막 `attemptId`를 기록한다. 제출 시 전체 CSV는 한 번 갱신하고, 인원별 CSV는 마지막 응시 ID가 변경된 인원만 다시 쓴다. 실제 브라우저 디스크 I/O는 실행 환경과 저장장치에 따라 편차가 커 Node 테스트 수치로 대체하지 않으며, 대신 `tests/report-file-storage.test.js`에서 변경되지 않은 인원 파일의 쓰기 횟수가 증가하지 않는 계약을 검증한다.
