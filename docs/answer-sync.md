# 답안 동기화 설계

## 1. 목표

- 입력 직후 화면 반응성을 유지한다.
- 일시적인 네트워크 장애에도 답안을 잃지 않는다.
- 재시도, 중복 요청, 응답 순서 변경이 답안을 과거 상태로 되돌리지 않게 한다.
- 사용자에게 실제 저장 상태를 명확히 알린다.

## 2. 클라이언트 상태

문제별로 다음 값을 관리한다.

| 필드 | 설명 |
| --- | --- |
| `questionId` | 문제 식별자 |
| `value` | 현재 화면 답안 |
| `clientRevision` | 해당 문제의 로컬 변경마다 증가하는 번호 |
| `serverRevision` | 마지막으로 서버가 수락한 번호 |
| `syncStatus` | `saved`, `dirty`, `saving`, `offline`, `failed`, `conflict` |
| `updatedAt` | 로컬 변경 시각, 표시·진단용이며 충돌 판정 기준은 아님 |

재사용과 디버깅이 필요하므로 현재 문제와 미전송 답안은 명시적인 컬렉션으로 유지한다. 예상 문제 수가 작아 복잡한 스트리밍이나 별도 인덱스 최적화는 도입하지 않는다.

## 3. 저장 흐름

1. 답안 변경을 메모리 상태에 즉시 반영한다.
2. 문제의 `clientRevision`을 증가시키고 상태를 `dirty`로 바꾼다.
3. 짧은 debounce 후 최신 변경을 저장 대기열에 넣는다.
4. 저장 요청에 응시 ID, 문제 ID, 답안, 클라이언트 리비전, 알려진 서버 리비전, 멱등성 키를 포함한다.
5. 서버가 응시 상태, 마감, 문제 소속, 답안 형식을 검증한다.
6. 서버가 새 리비전을 수락하면 저장 시각과 서버 리비전을 반환한다.
7. 응답의 클라이언트 리비전이 현재 리비전과 같을 때만 화면 상태를 `saved`로 바꾼다. 더 최신 로컬 변경이 있으면 `dirty` 상태를 유지한다.

## 4. API 예시

```http
PUT /api/v1/attempts/{attemptId}/answers/{questionId}
Idempotency-Key: 7c98...
Content-Type: application/json
```

```json
{
  "value": ["choice-2"],
  "clientRevision": 4,
  "knownServerRevision": 3
}
```

```json
{
  "questionId": "question-1",
  "clientRevision": 4,
  "serverRevision": 4,
  "savedAt": "2026-06-08T10:10:15.240Z",
  "serverTime": "2026-06-08T10:10:15.245Z"
}
```

## 5. 멱등성과 순서 보장

- 동일 멱등성 키의 요청은 같은 결과를 반환한다.
- 서버는 `knownServerRevision`이 현재 값과 일치할 때만 조건부 갱신한다.
- 과거 클라이언트 리비전이 늦게 도착해도 최신 서버 답안을 덮어쓰지 않는다.
- 문제별 저장은 독립적으로 처리하여 한 문제의 실패가 전체 답안 저장을 막지 않게 한다.
- 최종 제출은 별도의 제출 멱등성 키를 사용한다.

## 6. 오프라인과 재접속

- 네트워크 실패 시 최신 문제별 변경만 브라우저의 임시 대기열에 저장한다.
- 연결 복구 또는 페이지 재진입 시 먼저 서버의 응시 상태와 답안 리비전을 조회한다.
- 서버가 `IN_PROGRESS`이고 마감 전이면 미전송 변경을 문제별 순서로 재전송한다.
- 서버가 `SUBMITTED`, `INVALIDATED` 또는 마감 상태이면 로컬 변경을 전송하지 않고 사용자에게 결과를 안내한다.
- 민감한 답안의 장기 보관을 피하기 위해 임시 데이터는 응시 종결, 로그아웃, 보존 기한 만료 시 삭제한다.
- 브라우저 저장소 사용 여부와 암호화 수준은 위협 모델 및 지원 브라우저 확정 후 ADR로 결정한다.

## 7. 충돌 정책

| 상황 | 처리 |
| --- | --- |
| 동일 기기의 응답 순서 변경 | 높은 리비전만 유지 |
| 다른 탭에서 변경 | 서버 리비전 불일치로 거부 후 최신 답안 표시 및 재확인 |
| 다른 기기에서 변경 | 마지막 활성 쓰기 세션 정책에 따라 기존 기기를 읽기 전용 전환 |
| 마감 이후 미전송 답안 | 수락하지 않고 마지막 서버 저장본으로 자동 제출 |
| 이미 제출된 응시 | 기존 제출 결과 반환, 답안 변경 거부 |

단순한 “마지막 시각 우선”은 클라이언트 시계 조작과 오차에 취약하므로 사용하지 않는다.

## 8. 사용자 피드백

- `저장 중`: 서버 응답 대기
- `저장 완료 · HH:MM:SS`: 최신 화면 답안까지 서버가 수락
- `오프라인 · 연결 후 자동 저장`: 임시 대기열에 변경 존재
- `저장 실패 · 다시 시도`: 재시도 한도를 초과한 오류
- `다른 화면에서 변경됨`: 서버 리비전 충돌로 사용자 확인 필요
- `제출 완료`: 제출 식별자와 서버 제출 시각 표시
