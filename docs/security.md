# 보안 및 개인정보 기준

## 1. 보호 대상

- 시험 문제, 정답, 해설, 채점 기준
- 응시자 신원 및 배정 정보
- 진행 중 답안과 최종 제출물
- 시험 일정과 접근 정책
- 관리자 조치와 감사 기록
- 감독 기능을 도입할 경우 생성되는 영상·음성·화면 데이터

## 2. 핵심 위협과 통제

| 위협 | 통제 |
| --- | --- |
| 다른 응시 세션 접근(IDOR) | 모든 요청에서 인증 주체와 응시 소유권 재검사 |
| 정답·채점 기준 노출 | 응시자 DTO와 관리·채점 DTO 분리, 응답 테스트 |
| 세션 탈취 | 짧고 안전한 세션, 재인증, 쿠키 보호, TLS 강제 |
| 저장 요청 재전송 | 멱등성 키, 서버 리비전, 조건부 갱신 |
| 마감 이후 답안 변경 | 서버 시각과 응시 상태를 변경 트랜잭션에서 검사 |
| 중복 최종 제출 | 응시 ID 유일성 제약과 원자적 상태 전이 |
| 관리자 권한 오용 | 역할 분리, 최소 권한, 필수 사유, 감사 이벤트 |
| 클라이언트 코드 변조 | 클라이언트를 신뢰하지 않고 모든 규칙을 서버에서 검증 |
| 민감 정보 로그 유출 | 답안 원문, 토큰, 쿠키, 불필요한 개인정보 로그 금지 |

## 3. 인증과 세션

- 인증 방식은 운영 환경의 통합 인증 요구를 확인해 결정한다.
- 쿠키 기반 세션은 `HttpOnly`, `Secure`, 적절한 `SameSite` 설정을 사용한다.
- 상태 변경 요청에는 CSRF 방어를 적용한다.
- 로그인과 고위험 관리자 작업에 요청 속도 제한과 이상 징후 탐지를 적용한다.
- 중복 로그인 정책에 따라 쓰기 가능한 활성 클라이언트를 서버에서 식별한다.
- 비밀번호를 직접 관리한다면 검증된 적응형 해시와 계정 복구 보호를 사용한다.

## 4. 인가

인가 검사는 URL이나 화면 숨김이 아니라 서버의 애플리케이션 계층에서 수행한다.

1. 인증 여부 확인
2. 역할 또는 명시적 권한 확인
3. 대상 자원의 소유권·담당 범위 확인
4. 시험과 응시 상태 확인
5. 허용된 필드와 행위만 실행
6. 고위험 변경은 사유와 감사 이벤트 기록

관리자도 담당하지 않은 시험 데이터에는 기본적으로 접근할 수 없다.

## 5. 데이터 보호

- 전송 구간은 TLS를 사용한다.
- 저장 데이터 암호화는 운영 플랫폼의 관리형 암호화를 기본으로 한다.
- 비밀 값은 소스 코드나 일반 환경 파일에 커밋하지 않고 비밀 관리 시스템을 사용한다.
- 답안, 제출, 개인정보의 보존 기간과 파기 기준을 데이터 분류표로 관리한다.
- 백업에도 동일한 접근 통제와 보존·파기 정책을 적용한다.
- 운영 및 분석 데이터에는 가능한 한 가명 식별자를 사용한다.

## 6. 감사 이벤트

다음 이벤트를 최소한 기록한다.

- 로그인 성공·실패와 세션 강제 종료
- 시험 생성·수정·게시·취소
- 응시자 배정 변경
- 응시 시작과 재접속
- 답안 저장 실패·충돌·마감 후 거부
- 최종 제출과 자동 제출
- 응시 무효화·재응시 허용
- 점수 변경과 결과 공개
- 권한 변경과 감사 로그 조회

감사 이벤트에는 이벤트 유형, 행위자, 대상 ID, 서버 시각, 요청 추적 ID, 결과, 필요한 사유를 기록한다. 답안 원문과 인증 비밀은 기록하지 않는다.

## 7. 부정행위 방지 원칙

- 개발자 도구, 단축키, 복사·붙여넣기 차단은 우회 가능하므로 보안 경계로 간주하지 않는다.
- 문제·선택지 무작위화, 문제 세트 분리, 시간 정책, 이상 행위 기록을 우선 검토한다.
- 전체 화면 이탈 등 클라이언트 신호는 오탐 가능성이 있으므로 자동 제재가 아닌 참고 이벤트로 사용한다.
- 웹캠, 마이크, 화면 녹화는 법률·개인정보 검토, 명시적 동의, 대체 응시 수단, 보존 기간, 열람 권한, 삭제 절차를 확정한 뒤 별도 단계로 도입한다.

## 8. 보안 검증

- 소유권과 역할별 API 통합 테스트
- 응시자 응답에 정답 필드가 포함되지 않는 계약 테스트
- OWASP ASVS 기반 체크리스트와 의존성 취약점 검사
- 제출·마감·리비전 경쟁 조건 테스트
- 배포 전 비밀 스캔과 보안 헤더 검사
- 주요 출시 전 위협 모델 및 침투 테스트 검토
