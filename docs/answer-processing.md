# 로컬 답안 처리 설계

> 파일명은 기존 문서 링크 호환을 위해 유지하지만, 온라인 동기화 기능은 사용하지 않는다.

## 1. 목표

- 답안 입력 직후 화면에 반영한다.
- 네트워크 요청 없이 현재 실행 환경의 메모리에서 답안을 관리한다.
- 제출 시점의 답안을 고정하고 동일한 채점 규칙으로 결과를 계산한다.
- 사용자에게 응답 여부, 제출 상태, 채점 결과를 명확히 알린다.

## 2. 로컬 상태

문제별로 다음 값을 관리한다.

| 필드 | 설명 |
| --- | --- |
| `questionId` | 문제 식별자 |
| `value` | 현재 답안 |
| `answered` | 유효한 응답이 입력되었는지 여부 |
| `updatedAt` | 로컬 변경 시각, 화면 표시용 |

시험 전체에는 다음 상태를 둔다.

| 상태 | 설명 |
| --- | --- |
| `READY` | 시험지를 읽고 시작을 기다리는 상태 |
| `IN_PROGRESS` | 답안을 작성할 수 있는 상태 |
| `SUBMITTED` | 답안 스냅샷이 고정된 상태 |
| `GRADED` | 문항별 평가와 총점 계산이 끝난 상태 |

예상 문제 수가 작으므로 답안은 명시적인 메모리 컬렉션으로 유지한다. 재사용과 디버깅이 쉬운 단순 구조를 우선하며 별도 동기화 대기열이나 리비전은 두지 않는다.

## 3. 답안 입력 흐름

1. 응시자가 답안을 변경한다.
2. 문제 유형에 맞는 값인지 로컬에서 확인한다.
3. 답안 컬렉션의 해당 문제 값을 갱신한다.
4. 빈 값 여부를 계산해 문제 목록의 응답 상태를 갱신한다.
5. 화면 이동과 관계없이 시험이 열린 동안 메모리 상태를 유지한다.

브라우저 새로고침이나 애플리케이션 종료 복구는 MVP에서 보장하지 않는다. 필요성이 확인되면 명시적 임시 저장 기능을 별도 요구사항으로 추가한다.

## 4. 제출 흐름

1. 미응답 문항 수를 계산해 사용자에게 알린다.
2. 사용자가 최종 제출을 확인하거나 시간 만료 정책이 제출을 실행한다.
3. 현재 답안 컬렉션을 변경 불가능한 제출 스냅샷으로 복사한다.
4. 시험 상태를 `SUBMITTED`로 바꾸고 추가 답안 변경을 막는다.
5. 각 문항의 채점 규칙을 한 번 적용한다.
6. 문항별 결과를 합산해 총점을 계산하고 상태를 `GRADED`로 바꾼다.
7. 결과 화면을 표시하고 선택적으로 결과 파일 저장 또는 인쇄를 제공한다.

## 5. 채점 결과 모델

```json
{
  "examId": "exam-1",
  "submittedAt": "2026-06-08T10:10:15.240Z",
  "score": 80,
  "maxScore": 100,
  "items": [
    {
      "questionId": "question-1",
      "status": "correct",
      "earnedScore": 10,
      "maxScore": 10
    }
  ]
}
```

`submittedAt`은 결과 표시와 파일 정렬을 위한 로컬 시각이다. 엄격한 시간 증명이나 서버 권위 시각을 제공하지 않는다.

## 6. 문제 유형별 평가

| 유형 | 평가 방식 |
| --- | --- |
| 단일 선택 | 응답 보기 ID와 정답 보기 ID의 일치 여부 |
| 복수 선택 | 응답 집합과 정답 집합의 완전 일치 여부 |
| 단답형 | 시험지에 정의된 정규화 후 허용 답안과 비교 |
| 수동 검토 | 자동 점수를 확정하지 않고 `review_required` 표시 |

단답형 정규화는 앞뒤 공백 제거, 연속 공백 축소, 대소문자 무시처럼 명시된 규칙만 사용한다. 의미 추론이나 외부 AI 호출은 하지 않는다.

## 7. 사용자 피드백

- `응답 완료`: 현재 문항에 유효한 답안이 있음
- `미응답`: 답안이 비어 있음
- `제출 확인`: 미응답 문항 수와 제출 후 수정 불가 안내
- `채점 중`: 로컬 채점 처리 중
- `채점 완료`: 총점과 문항별 결과 표시
- `검토 필요`: 자동 채점할 수 없는 문항 표시
- `파일 오류`: 시험지 구조나 채점 기준이 잘못되어 시작할 수 없음
