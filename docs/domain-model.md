# 도메인 모델

## 1. 핵심 엔터티

| 엔터티 | 주요 속성 | 책임 |
| --- | --- | --- |
| `Exam` | ID, 제목, 입장 기간, 제한 시간, 상태, 정책 버전 | 시험 일정과 적용 정책 관리 |
| `Question` | ID, 시험 ID, 유형, 본문, 배점, 순서 | 응시자에게 제공할 문제 정의 |
| `Choice` | ID, 문제 ID, 표시 내용, 순서 | 객관식 선택지 정의 |
| `Assignment` | 시험 ID, 사용자 ID, 응시 가능 여부 | 시험과 응시자 연결 |
| `Attempt` | ID, 시험 ID, 사용자 ID, 상태, 시작·마감·제출 시각, 정책 스냅샷 | 한 번의 응시 생명주기 관리 |
| `Answer` | 응시 ID, 문제 ID, 값, 서버 리비전, 저장 시각 | 문제별 최신 수락 답안 관리 |
| `Submission` | ID, 응시 ID, 제출 원인, 제출 시각, 답안 스냅샷 해시 | 변경 불가능한 최종 제출 기록 |
| `AuditEvent` | 행위자, 대상, 이벤트 유형, 시각, 사유, 메타데이터 | 주요 행위의 추적 가능성 제공 |

정답과 채점 기준은 응시자용 `Question` 조회 모델과 분리된 저장 및 권한 경계를 사용한다.

## 2. 응시 상태

- `CREATED`: 배정은 확인했으나 시험을 시작하지 않은 상태
- `IN_PROGRESS`: 답안 조회·저장이 가능한 상태
- `SUBMITTING`: 최종 제출 트랜잭션이 진행 중인 상태
- `SUBMITTED`: 최종 제출이 확정된 상태
- `EXPIRED`: 시간 만료가 확인되었으나 제출 확정이 복구 작업을 기다리는 상태
- `INVALIDATED`: 관리자가 사유를 기록하고 무효화한 상태

## 3. 상태 전이

| 현재 상태 | 이벤트 | 다음 상태 | 조건 |
| --- | --- | --- | --- |
| `CREATED` | 시험 시작 | `IN_PROGRESS` | 배정 유효, 입장 가능 시간, 활성 응시 중복 없음 |
| `IN_PROGRESS` | 최종 제출 요청 | `SUBMITTING` | 서버 시각이 마감 전이며 제출 권한 보유 |
| `IN_PROGRESS` | 시간 만료 감지 | `EXPIRED` | 서버 시각이 마감 이상 |
| `SUBMITTING` | 제출 확정 | `SUBMITTED` | 답안 스냅샷과 제출물이 트랜잭션으로 저장됨 |
| `SUBMITTING` | 복구 가능한 실패 | `IN_PROGRESS` | 마감 전이고 제출물이 생성되지 않음 |
| `EXPIRED` | 자동 제출 확정 | `SUBMITTED` | 마지막 서버 저장 답안으로 제출물 생성 |
| `CREATED`/`IN_PROGRESS` | 관리자 무효화 | `INVALIDATED` | 권한과 필수 사유가 확인됨 |

`SUBMITTED`와 `INVALIDATED`는 종결 상태다. 금지된 상태 전이는 서버에서 거부하고 감사 이벤트를 남긴다.

## 4. 시간 규칙

- 모든 영속 시각은 UTC로 저장한다.
- 시험 시작과 마감 판정은 서버 시각만 신뢰한다.
- `started_at`은 최초 시작 성공 시 한 번만 기록한다.
- `deadline_at`은 응시 시작 시 정책 스냅샷으로 계산한다.
- 기본 계산식은 `min(started_at + duration, exam.entry_closes_at)`이다.
- 화면 타이머는 응답의 `server_time`과 `deadline_at`을 사용해 클라이언트 시계 오차를 보정한다.
- 브라우저 타이머는 표시 보조 수단이며 저장·제출 허용 여부를 결정하지 않는다.

## 5. 불변 조건

1. 사용자·시험 조합에는 동시에 하나의 활성 `Attempt`만 존재한다.
2. `Attempt`마다 최종 `Submission`은 최대 하나만 존재한다.
3. `SUBMITTED` 이후에는 `Answer`를 변경할 수 없다.
4. `Answer.server_revision`은 문제별로 단조 증가한다.
5. `deadline_at` 이후의 새 답안은 수락하지 않는다.
6. 제출물은 제출 시점의 서버 저장 답안 스냅샷을 참조하거나 보존한다.
7. 관리자 수정은 행위자와 사유 없이 수행할 수 없다.

데이터베이스의 유일성 제약, 조건부 갱신, 트랜잭션을 함께 사용해 애플리케이션 오류나 동시 요청에서도 불변 조건을 유지한다.

## 6. 제출 트랜잭션

1. 응시 행을 잠그거나 상태와 리비전을 조건으로 원자적 갱신한다.
2. 현재 서버 시각과 응시 상태를 검사한다.
3. 상태를 `SUBMITTING`으로 전환한다.
4. 최신 서버 답안 목록과 해시를 생성한다.
5. 유일한 제출물을 생성한다.
6. 응시 상태와 `submitted_at`을 `SUBMITTED`로 확정한다.
7. 제출 감사 이벤트를 같은 트랜잭션에 기록한다.

동일 멱등성 키 또는 동일 응시 ID로 요청이 반복되면 기존 제출 결과를 반환한다.
